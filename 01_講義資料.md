# 講義資料：保守性を高める開発技術の基礎

---

## セクション1：オリエンテーション（20分）

### 1-1. SE保守・運用 vs SE開発 — 何が違うのか？

| 観点 | SE保守・運用 | SE開発 |
|---|---|---|
| 主な仕事 | 既存システムの監視・障害対応・パッチ適用 | 新機能の設計・実装・テスト |
| コードへの向き合い方 | 動いているコードを壊さない | 将来変えやすいコードを書く |
| 変更の粒度 | 最小限の変更（リスク回避） | 積極的な改善（品質向上） |
| 評価軸 | 障害件数・稼働率 | 開発速度・コード品質 |

**転向で最も問われるスキル**: 「動けばいい」から「変えやすく・読みやすく・テストしやすい」への思考転換

---

### 1-2. なぜ保守性・拡張性が重要か？

ソフトウェアのコストの **70〜80%** はリリース後のメンテナンスコストといわれている。

```
読みにくいコード → 修正に時間がかかる → バグを埋め込む → 障害 → 信頼失墜
```

**保守性の高いコードがもたらす効果**
- 新メンバーがすぐに理解できる（オンボーディングコスト削減）
- バグの影響範囲が限定される（障害の局所化）
- 仕様変更に素早く対応できる（ビジネス価値向上）

---

### 1-3. リファクタリングとは？

> **外部から見た振る舞いを変えずに、内部構造を改善すること**

- やること: コードの整理・命名改善・重複排除・責務分離
- やらないこと: 新機能追加・バグ修正（別のタスク）

**リファクタリングのタイミング**
1. 新機能追加の前（理解しやすくしてから追加する）
2. バグ修正の後（根本原因を潰してから整理する）
3. コードレビューで指摘を受けたとき

## セクション2：リファクタリング基礎講座（30分）

### 2-1. DRY原則（Don't Repeat Yourself）

> **「すべての知識はシステム内で単一・明確・権威ある表現を持つべき」**

**悪い例（重複コード）**
```python
# 注文処理でも、返品処理でも、ステータス確認でも同じ顧客取得ロジックが散在
def process_order(cid):
    if cid == 1:
        name = "田中太郎"
        email = "tanaka@example.com"
    elif cid == 2:
        name = "鈴木花子"
        email = "suzuki@example.com"
    # ...

def process_return(cid):
    if cid == 1:        # ← まったく同じロジックが再登場
        name = "田中太郎"
        email = "tanaka@example.com"
    # ...
```

**良い例（単一化）**
```python
CUSTOMERS = {
    1: {"name": "田中太郎", "email": "tanaka@example.com"},
    2: {"name": "鈴木花子", "email": "suzuki@example.com"},
}

def get_customer(customer_id: int) -> dict | None:
    return CUSTOMERS.get(customer_id)

# 以降はどこでも get_customer() を呼ぶだけ
```

**DRY違反が招く問題**: 顧客データを修正する際に、コピーされた全箇所を見つけて直す必要がある → 1箇所修正漏れでバグ発生

---

### 2-2. SOLID原則（概要）

| 原則 | 略称 | 意味 | 一言まとめ |
|---|---|---|---|
| Single Responsibility Principle（単一責任の原則） | S | クラス・関数は1つの責務だけ持つ | 「1つのことだけやる」 |
| Open/Closed Principle（開放閉鎖の原則） | O | 拡張に開き、修正に閉じる | 「既存コードを変えずに機能追加できる」 |
| Liskov Substitution Principle（リスコフの置換原則） | L | 派生クラスは基底クラスの代わりに使える | 「サブクラスは親の約束を守る」 |
| Interface Segregation Principle（インタフェース分離の原則） | I | 不要なインタフェースに依存させない | 「使わないメソッドを押しつけない」 |
| Dependency Inversion Principle（依存性逆転の原則） | D | 上位モジュールは下位実装に依存しない | 「抽象に依存する」 |

**本日の演習で特に意識してほしい原則: S（単一責任）と O（開放閉鎖）**

**SRP違反の例**
```python
# process_order が「合計計算」「割引適用」「送料計算」「メール送信」「在庫更新」を全部やっている
def process_order(customer_id, items, discount_code):
    # 合計計算
    total = sum(...)
    # 割引適用
    if discount_code == "GOLD10": total *= 0.9
    # 送料計算
    if total < 3000: shipping = 500
    # メール送信
    print(f"メール送信先: {email}")
    # 在庫更新
    for item in items: print(f"在庫を減らします")
    # ↑ 1関数が5つの責任を持っている = 変更理由が5つある
```

### 2-3. 命名規則・コメントのベストプラクティス

**命名の原則**

| NG | OK | 理由 |
|---|---|---|
| `c` | `customer` | 省略は読む速度を下げる |
| `t` | `subtotal` | 何のtotalか不明 |
| `ft` | `final_total` | 2単語で意味が明確に |
| `dc` | `discount_code` | コンテキストなしでは意味不明 |
| `p`, `q` | `price`, `quantity` | 汎用すぎる1文字変数 |

**良いコメントとは？**
```python
# NG: コードを読めばわかることを書いている
total = total * 0.9  # totalに0.9をかける

# OK: 「なぜ」を書いている
total = total * 0.9  # ゴールド会員は10%割引（2026年料金規程§3.2）

# NG: 古くなると嘘をつくコメント
# 割引コードはGOLD10のみ対応
if discount_code == "GOLD10": ...
elif discount_code == "SILVER5": ...  # ← 増えているのにコメントが古い
```

---

### 2-4. コード臭（Code Smell）の典型例

| Code Smell | 説明 | 対処法 |
|---|---|---|
| **重複コード** | 同じロジックが複数箇所にある | 関数・定数として抽出する |
| **長すぎる関数** | 1関数が100行を超えている | 責務ごとに分割する |
| **意味不明な命名** | `a`, `tmp`, `data2` など | 意図が伝わる名前にする |
| **マジックナンバー** | コード中に意味不明な数値が直書き | 名前付き定数（定義）にする |
| **コメントアウトコード** | 使わないコードが残っている | 削除する（git履歴に残る） |
| **巨大クラス** | 1クラスに何でも詰め込む | 責務ごとにクラスを分ける |
| **条件文の爆発** | `if/elif`が10個以上連なる | 辞書・ポリモーフィズムで置き換える |

---

### 2-5. リファクタリング手順（安全に進めるために）

1. **テストを書く**（変更前の動作を保証する）
2. **小さく変更する**（一度に大きく変えない）
3. **動作確認する**（テストを実行して壊れていないか確認）
4. **コミットする**（変更履歴を細かく残す）
5. **繰り返す**

> **「テストなきリファクタリングはリスクでしかない」**  
> — Martin Fowler, *Refactoring* より

## セクション4：ペアレビュー（20分）

### レビューの進め方
1. 相手のGoogleドキュメントを開き、改善方針を確認する（2分）
2. 観点チェックリストに沿って良い点・改善提案を記入する（10分）
3. お互いのコメントを読み合わせ・議論する（8分）

### レビューコメントの書き方

**良いレビューコメントの例**
```
✅ 良い点: get_customer() として抽出したことで、DRY原則が守られています。
   将来的に顧客データをDBから取得する変更も1箇所の修正で済みます。

💡 改善提案: calculate_subtotal() の中でカテゴリごとに条件分岐していますが、
   全カテゴリで同じ計算をしているため、条件分岐は不要では？
   `sum(item["price"] * item["quantity"] for item in items)` と書けそうです。
```

**避けるべきレビューコメント**
```
❌ 「このコードはひどい」（人格攻撃）
❌ 「なんでこう書いたの？」（非難）
❌ 「私ならこう書く」だけで終わり（理由がない）
```

> レビューはコードを改善するもの。書いた人を評価するものではない。

## セクション6：変更影響分析ケーススタディ（20分）

### ケース：仕様変更依頼

> **依頼**: 「割引コードを新たに `NEW30`（30%OFF）追加してほしい。また、送料の無料ラインを5,000円から8,000円に引き上げてほしい。」

`04_変更影響チェックリスト.md` を使って、影響範囲を洗い出してください。

### 変更影響の考え方

```
1. 変更する対象を特定する
   → どの関数・クラス・定数が直接変わるか？

2. 呼び出し元（上流）を調べる
   → 変更した関数を呼んでいるのはどこか？

3. 呼び出し先（下流）を調べる
   → 変更した関数が依存しているものは何か？

4. テストへの影響を確認する
   → どのテストケースが失敗するか？

5. 外部インタフェースへの影響を確認する
   → APIのリクエスト・レスポンスが変わるか？ドキュメントは？
```

### リファクタリング前後の比較

**リファクタリング前（問題コード）での変更コスト**
- 割引コード追加: `process_order` の `if/elif` チェーンに1行追加
- 送料ルール変更: マジックナンバーを探して修正 → 見落としリスクあり

**リファクタリング後（模範解答）での変更コスト**
```python
# 割引コード追加: 辞書に1行追加するだけ
DISCOUNT_RATES = {
    "GOLD10": 0.10,
    "SILVER5": 0.05,
    "SUMMER20": 0.20,
    "WINTER15": 0.15,
    "NEW30": 0.30,  # ← これだけ
}

# 送料ルール変更: 定数テーブルの数値を変えるだけ
SHIPPING_RULES = [
    (8000, 0),    # ← 5000 → 8000 に変更
    (3000, 300),
    (0, 500),
]
```
